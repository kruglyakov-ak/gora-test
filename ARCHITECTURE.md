1. Какое решение для стейт-менеджмента вы выбрали и почему именно его, а не
   альтернативы?

Выбрал Zustand.
Причины выбора:

Для небольшого проекта (один экран) Zustand предлагает лучший баланс между простотой и функциональностью.
Из коробки поддерживает persist (middleware), что идеально закрыло требование о сохранении истории сообщений после перезагрузки страницы.
Минимальный boilerplate, отличная интеграция с TypeScript и удобная работа с асинхронными действиями.
По сравнению с Redux / Redux Toolkit — значительно меньше кода и сложности. По сравнению с Jotai — более привычный и «цельный» API для управления одним большим хранилищем (чат).

Альтернативы (Redux Toolkit, Jotai, MobX) были рассмотрены, но показались либо избыточными, либо менее удобными для данного объёма задачи.

2. Как вы разделили слои приложения? Покажите структуру папок.

Придерживался Feature-Sliced Design (FSD). Даже для тестового задания это позволило сохранить чёткую структуру, масштабируемость и удобство навигации по коду.
Структура проекта:
Bashsrc/
├── app/ # Инициализация приложения, глобальные стили
├── features/
│ └── chat/ # Основная и единственная фича
│ ├── api/ # API-слой (моки)
│ ├── model/ # Состояние, бизнес-логика, types
│ ├── lib/ # Вспомогательные утилиты фичи
│ ├── ui/ # Все UI-компоненты
│ └── index.ts # Public API фичи
├── pages/ # Страницы приложения
├── shared/ # Переиспользуемые утилиты и хуки
└── ARCHITECTURE.md
Такое разделение позволяет легко добавлять новые фичи в будущем без нарушения структуры.

3. Что бы вы сделали иначе, если бы это был production-проект с командой из 5
   человек?

Перешёл бы на строгий FSD с обязательным Public API для каждого слайса и строгим контролем зависимостей.
Внедрил бы git-flow (или trunk-based development) с отдельными ветками для фич, релизов и hotfix-ов.
Добавил бы автоматизированное тестирование: unit-тесты (Vitest), компонентные тесты (React Testing Library) и E2E-тесты.
Создал бы собственный UI-kit + Storybook вместо использования готовых библиотек (DaisyUI, lucide и т.д.).
Убрал бы все сторонние UI-компоненты (TextareaAutosize, EmojiPicker) и заменил бы их на собственные кастомные.
Добавил бы полноценную тёмную/светлую тему через CSS-переменные или Tailwind.
Заменил бы моки на реальный бэкенд с WebSocket (или Socket.io) вместо setTimeout.
Подключил бы Sentry для мониторинга ошибок.
Добавил бы i18n (react-i18next) с самого начала.
Настроил бы CI/CD, lint-staged, husky, conventional commits и preview-деплои.
